# 백엔드 도입 가이드 — 엘드리아 연대기

현재 게임은 **완전 오프라인/로컬**로 동작합니다(세이브는 기기 저장소, 아레나·길드는 시뮬레이션 상대).
이 문서는 (1) 백엔드가 필요한 영역, (2) 필요 없는 영역, (3) 셋팅 방법·단계·비용을 정리합니다.

---

## 0. 핵심 전제 — 우리 구조의 강점

- **코어(`system/core/*.mjs`)는 의존성 없는 순수 ESM**입니다.
  → 같은 코드를 **Node 백엔드에서 그대로 import** 해 전투를 재-시뮬레이션할 수 있습니다.
  → 서버 권위적 검증(안티치트)·비동기 PvP 결과 재현이 매우 저렴합니다.
- **세이브 스키마 버전(`SAVE_VERSION`)** 과 정규화(`normalize`)가 이미 있어, 서버 저장/마이그레이션에 바로 활용 가능합니다.
- **이관 코드(`exportCode`/`importCode`)** 가 이미 있어, 계정 연동 전 임시 백업 경로로 씁니다.

원칙: **코어 루프는 클라이언트에 유지**하고, 서버는 "저장·검증·랭킹·매칭·결제·운영"만 담당(하이브리드).

---

## 1. 백엔드가 필요한 영역 (우선순위)

| 우선 | 영역 | 왜 서버가 필요한가 | 현재 상태 |
|------|------|--------------------|-----------|
| ★★★ | **계정/인증** | 기기 분실·교체 시 진행 보존, 밴/제재, 다기기 | 로컬 + 이관코드만 |
| ★★★ | **클라우드 세이브** | 서버 저장·복구, 변조 방지 | 로컬 저장만 (`save.mjs`) |
| ★★★ | **IAP 영수증 검증** | 결제 위조·환불 어뷰징 차단 | 어댑터 골격만 (`iap.mjs`) |
| ★★☆ | **리더보드/랭킹** | 아레나 시즌·탑 랭킹, 명예 경쟁 | 없음(로컬 포인트만) |
| ★★☆ | **실서버 PvP 매칭** | 실제 유저 팀 스냅샷 매칭 | 시뮬 상대(`arena.mjs`) |
| ★★☆ | **원격 설정/운영** | 밸런스 핫픽스·이벤트·공지·우편함, 앱 재배포 없이 | 없음 |
| ★☆☆ | **길드/소셜** | 실제 길드원·채팅·협동 레이드 | 시뮬(`guild`) |
| ★☆☆ | **안티치트** | 세이브 변조·불가능 상태 차단 | 없음(클라 신뢰) |
| ★☆☆ | **푸시 알림(FCM)** | 복귀 유도·이벤트 안내 | 없음 |
| ★☆☆ | **분석/텔레메트리** | 리텐션·이탈·결제 지표 | 없음 |

## 2. 백엔드가 **필요 없는** 영역 (로컬 유지)

- 방치 틱·전투 판정·성장·환생 (`system/genres`, `system/core/resolution.mjs`)
- 가챠·장비·인챈트·룬·펫·유물·엠블럼·정령 (전부 클라 결정론)
- 출석·일일미션·던전(로컬 시계 기준)
- 코스메틱·프로필·설정

→ 이들은 서버로 옮기면 오히려 응답성·오프라인성을 해칩니다. **그대로 로컬 유지**.

## 3. 셋팅 방법 — 백엔드 옵션 비교

솔로/소규모 + Expo(RN) 기준 현실적인 선택지입니다.

| 옵션 | 강점 | 적합 영역 | 러닝커브 | 비용(무료 티어) |
|------|------|-----------|----------|-----------------|
| **Firebase** (Google) | 인증·문서DB·함수·리모트컨피그·FCM 올인원, Expo 친화 | 계정·클라우드세이브·푸시·원격설정·우편함 | 낮음 | 넉넉(Spark) |
| **PlayFab** (MS) | 게임 특화: 로그인·세이브·리더보드·상점·IAP검증·이벤트 | 랭킹·경제·IAP·라이브옵스 | 중 | 게임 특화 무료 |
| **Nakama** (Heroic Labs, OSS) | 실시간 매칭·소셜·길드채팅·리더보드, self-host 가능 | 실 PvP·길드·실시간 | 중상 | self-host 무료 |
| **Supabase** (OSS) | Postgres + Auth + Realtime, SQL 선호 시 | 계정·세이브·랭킹(SQL) | 중 | 넉넉 |

### 기능별 권장 조합
- **계정 + 클라우드 세이브 + 원격설정 + 푸시** → **Firebase** (가장 빠름)
- **리더보드 + IAP 검증 + 라이브옵스** → **PlayFab**
- **실시간 PvP/길드채팅** → **Nakama**

> 솔로 추천: **Firebase 단독으로 시작**(1~2단계 커버) → 랭킹/실PvP가 필요해지면 PlayFab 또는 Nakama 추가.

## 4. 단계별 도입 로드맵

### Phase 1 — 기반 (계정·세이브·결제·원격설정) · **가장 높은 ROI**
1. Firebase 프로젝트 생성 (Auth / Firestore / Functions / Remote Config / FCM)
2. Expo 연동
   - 관리형 워크플로: **Firebase JS SDK** (Auth·Firestore·RemoteConfig 웹 SDK)
   - 또는 dev client: `@react-native-firebase/*`
3. **익명 로그인** 먼저 → 나중에 Google/Apple 계정 연동(진행 보존)
4. Firestore 유저 문서: `users/{uid} = { save, saveVersion, updatedAt, checksum }`
   - `save.mjs`의 `serialize`/`deserialize` 결과 blob을 그대로 저장
   - 충돌 해결: `updatedAt` + 버전 비교, 최신 우선(또는 병합 규칙)
5. **Cloud Function**으로 **IAP 영수증 검증**(Google Play / App Store) 후 `shop.purchase` 보상 승인
6. **Remote Config**로 밸런스 계수(`balance.mjs`의 일부)·이벤트 배너·공지 원격화

### Phase 2 — 경쟁 (랭킹·비동기 PvP)
7. 리더보드: 아레나 시즌 포인트·탑 최고층 (PlayFab Leaderboard 또는 Firestore 집계 함수)
8. **비동기 PvP**: 유저가 "방어 팀 스냅샷 + 전투력"을 업로드 → 서버가 **같은 리그 풀**에서 상대 추출
   - 결과 판정은 **코어 `resolve()`를 서버(Node)에서 재사용** → 클라·서버 동일 결과, 안티치트 무료
   - 리그/약자보호 규칙(`arena.mjs`)을 서버 매칭에 그대로 이식

### Phase 3 — 소셜 (실 길드·채팅·협동)
9. Nakama 도입: 실 길드·길드채팅·실시간 협동 보스, 친구
10. 시즌/이벤트 라이브옵스, 우편함(보상 지급) 고도화

## 5. 아키텍처 원칙 (안티치트 포함)

- **서버 권위적 최소화**: 코어 계산은 클라, 서버는 검증·저장·랭킹·매칭·결제만.
- **결과가 아니라 상태를 저장**: 세이브 blob + 버전 + 체크섬. 서버는 업로드 시
  - 스키마/버전 검증(`normalize`),
  - **상한/합리성 검증**(예: 재화·레벨·전투력이 진행도 대비 불가능하게 크면 플래그),
  - 필요 시 **코어 재-시뮬레이션**으로 신고/랭킹 상위만 정밀 검증.
- **PvP는 비동기 스냅샷**: 실시간 동기화 없이 팀 데이터만 교환 → 서버 부하·구현 최소.
- **세이브 마이그레이션**: `SAVE_VERSION` 올릴 때 서버·클라 양쪽 `normalize`로 무중단.

## 8. 비용 계산 · 접속자수별 서버비 · 손익분기점

### 8.1 세이브 크기 (실측)
| 구간 | 크기 | 비고 |
|------|------|------|
| 신규(튜토리얼 직후) | ~1.8 KB | |
| 중반(캐릭 15·파티 완성) | ~22 KB | 대다수 활성 유저 |
| 엔드게임(캐릭 40 풀장착) | ~195 KB (gzip ~68 KB) | 상위 소수 |

→ 유저 1명 세이브는 **문서 1개**로 저장(Firestore 1MiB 한도 내). 평균 blob **~30 KB**로 비용 산정.

### 8.2 가정 (방치형 보수적)
- 유저당/일: 읽기 15, 쓰기 10, blob 30 KB
- Firestore 단가(대략): 읽기 $0.06/10만 · 쓰기 $0.18/10만 · 저장 $0.18/GB·월 · 전송 $0.12/GB
- 무료(일일): 읽기 5만·쓰기 2만, 전송 10GB/월. **인증·푸시·리모트컨피그는 사실상 무료**

### 8.3 접속자수(DAU)별 월 서버비
| DAU | 월 서버비(USD) | ≈ 원화 |
|-----|----------------|--------|
| 100 | $0 | ₩0 (무료 티어) |
| 1,000 | $0.4 | ₩600 |
| 2,000 | $2 | ₩2,800 |
| 5,000 | $9 | ₩12,000 |
| 10,000 | $21 | ₩29,000 |
| 50,000 | $119 | ₩160,000 |
| 100,000 | $241 | ₩325,000 |

> **DAU 1,000까지는 사실상 무료**(무료 일일 한도 안). 방치형은 서버 부하가 낮아 매우 저렴.

### 8.4 월 순매출(스토어 30% 차감 후) — ARPDAU 시나리오
| ARPDAU | DAU 1,000 | DAU 10,000 | DAU 100,000 |
|--------|-----------|------------|-------------|
| $0.02(저) | $420 | $4,200 | $42,000 |
| $0.05(중) | $1,050 | $10,500 | $105,000 |
| $0.10(고) | $2,100 | $21,000 | $210,000 |

### 8.5 손익분기점
**(A) 서버비 회수 BEP — 사실상 즉시.**
DAU 10,000·ARPDAU $0.02에서 순매출 $4,200 vs 서버 $21 → 서버비는 순매출의 **0.5%**.
유료 결제 유저 1~2명이면 한 달 서버비를 덮습니다. **서버비는 손익을 좌우하지 않습니다.**

**(B) 실질 BEP — 서버가 아니라 "유저 획득비(UA)와 개발 시간"이 결정.**
- 오가닉(무료) 유입: IAP가 조금이라도 발생하면 **즉시 흑자**(서버비 미미).
- 유료 광고 유입: **LTV > CPI** 여야 흑자.
  - LTV ≈ ARPDAU × 평균 잔존일수 × 0.7(스토어차감)
    - 예) ARPDAU $0.05 · 30일 → LTV ≈ $1.05
    - 예) ARPDAU $0.10 · 45일 → LTV ≈ $3.15
  - CPI(설치당 광고비) ≈ $0.5~$3 → **LTV가 CPI보다 커야 광고 집행이 남는 장사**.

### 8.6 결론
1. **서버비는 걱정거리가 아님** — 수천 DAU까지 무료, DAU 1만이어도 월 ₩3만 수준.
2. **진짜 원가는 ① 스토어 수수료 30% ② 개발 시간 ③ (광고 집행 시) UA 비용**.
3. 권장: **오가닉 중심으로 출시 → 서버비 무료 구간에서 IAP 검증 → LTV 측정 후에만 유료 UA**.
4. 백엔드 착수는 비용이 아니라 **작업 시간** 관점에서 계획(Phase 1 MVP는 무료 티어 안에서 전부 가능).
