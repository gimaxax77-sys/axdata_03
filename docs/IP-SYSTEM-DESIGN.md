# IP 게임 시스템 설계서

## 0. 한 줄 정의

> **IP = 게임 시스템(엔진) 그 자체.**
> 시스템은 고정 자산이고, 그 위에 **장르(RPG↔방치형)** 와 **컨셉(테마)** 만 갈아끼워
> 여러 게임으로 운영한다.

캐릭터·세계관이 IP가 아니다. **성장·전투·경제 규칙**이 IP다.
따라서 한 번 잘 만든 시스템을 판타지 RPG로도, SF 방치형으로도 재출시할 수 있다.

---

## 1. 3-레이어 아키텍처

```
┌─ Core System (IP · 고정) ───────────────────────────┐
│  장르도 컨셉도 모른다. 순수 규칙과 데이터.               │
│    · archetypes  유닛 역할(방어/공격/지원)              │
│    · stats       레벨·랭크 성장 + 모디파이어 반영         │
│    · modifiers   성장요소 합산(스킬·강화·장비)           │
│    · skills      스킬 카탈로그/장착/강화                 │
│    · enhance     강화(각인) 노드 정의                    │
│    · gear        장비 카탈로그/제작/장착/강화             │
│    · gacha       소환(확률·천장) — RNG 주입              │
│    · rng         시드 가능 난수                          │
│    · character   레벨업·돌파·장착·강화 액션(자원 소모)    │
│    · units       유닛 인스턴스 → 전투 프로필              │
│    · resolution  전투 판정 엔진  → { win, duration }    │
│    · progression 무한 스테이지 사다리                    │
│    · economy     추상 자원(currency/growth/summon)     │
│    · gameState   세이브(보유·진행·지갑) — 장르/컨셉 없음  │
├─ Genre Layer (Driver · 스왑) ───────────────────────┤
│  "시간을 누가 굴리나" 만 다르다. 같은 resolve()를 쓴다.   │
│    · rpg   플레이어가 battle() 트리거, 행동력 소모, 능동   │
│    · idle  tick(dt) 자동 반복·누적, 오프라인·환생, 수동0  │
├─ Concept Layer (Skin · 스왑) ───────────────────────┤
│  숫자는 그대로. 이름·이모지·색상 + 캐릭터 도감(정체성).     │
│    · fantasy  VANGUARD→수호기사, 카엘/루나… (roster)      │
│    · scifi    VANGUARD→가디언프레임, 유닛-07/노바…          │
└─────────────────────────────────────────────────────┘
```

의존 방향은 **한 방향뿐**: `concept → genre → core`.
core 는 위 레이어를 절대 import 하지 않는다 → 그래서 재사용된다.

## 2. 핵심 설계: 판정 엔진이 두 장르를 잇는다

`resolution.resolve(party, challenge)` 는 결정론적으로 다음을 반환한다:

```js
{ win: boolean, duration: number /* 클리어 소요 초 */ }
```

- **RPG** 는 `win` 을 쓴다 → 이기면 다음 스테이지 개방(진행 게이팅).
- **방치형** 은 `duration` 을 쓴다 → `dt / duration` 으로 초당 클리어 수·보상 누적.

> 하나의 순수 함수가 능동형과 자동형을 동시에 지탱한다. 이게 이 IP의 심장이다.

판정 모델(요약): 각 진영을 `HP · 유효DPS` 로 환산해 상호 처치 시간을 비교.
방어(def)는 `100/(100+def)` 피해 감쇠, 지원형은 팀 공격력 버프를 더한다.
스킬 효과(치명타·흡혈·관통·팀버프)도 여기서 반영된다(아래 참조).

## 3. 캐릭터 성장 — "같은 캐릭터를 다르게"

같은 원형(archetype)·같은 레벨이어도 **어떻게 키우느냐**로 유닛을 분화시킨다.
세 축이 서로 다른 종류의 차별화를 만든다:

| 축 | 성격 | 자원 | 효과 |
|---|---|---|---|
| **레벨업** | 균일 성장 | growth | 전 스탯 +8%/레벨. 상한 = 랭크×20 |
| **돌파(랭크)** | 한계 개방 | summon | 스탯 +25%/랭크 + 레벨 상한↑ + 스킬 슬롯↑ |
| **스킬 장착** | 선택 분화 | (장착 무료) | 슬롯에 스킬을 골라 끼움 → 빌드/역할 |
| **스킬 강화** | 선택 심화 | growth | 스킬 레벨 +15%/레벨 |
| **강화(각인)** | 방향 투자 | currency | 특정 스탯 노드에 집중(atk/hp/def/crit) |
| **장비** | 착용 성장 | currency | 슬롯별 flat 스탯 + 효과, 강화로 +12%/레벨 |

### 3-1. 모디파이어 파이프라인 (`modifiers.mjs`)

성장 요소가 흩어져 있으면 스탯 계산이 지옥이 된다. 그래서 **모든 소스를 한 곳에서 합산**한다:

```
collectUnitModifiers(unit)  →
  { statPct: {atk,hp,def,spd},                  // % 스탯 증가
    statFlat:{atk,hp,def,spd},                  // 고정 스탯 (장비)
    effect:  {critChance,critDamage,lifesteal,defPierce},  // 전투 효과
    teamBuff:{atk} }                             // 팀 버프
```

소스는 4개: **원형 버프 · 장착 스킬(레벨 반영) · 강화 노드 · 장착 장비**.
`stats.mjs`(스탯)와 `resolution.mjs`(판정)가 이 합산 결과를 공유한다.
새 성장 요소(예: 유물/각성)를 붙여도 여기 소스만 추가하면 된다.

### 3-2. 스탯 계산 순서 (`stats.mjs`)

```
1) 기본스탯 × 레벨배수 × 랭크배수      (원형 성장 — 균일)
2) × (1 + statPct)                    (스킬·각인 — 방향성)
3) + statFlat                         (장비 — 착용)
```

### 3-3. 스킬 전투 효과가 판정에 들어가는 법

| 효과 | 판정 반영 |
|---|---|
| 치명타 crit | dps × (1 + 확률×치명피해) — 프로필 단계 |
| 흡혈 lifesteal | 파티 실효HP × (1+흡혈), 상한 60% |
| 관통 defPierce | 적 방어 × (1−관통), 파티 내 최댓값 |
| 팀버프 teamBuff | 파티 dps × (1+합산) |

### 3-4. 검증 (`node system/demo-build.mjs`)

동일한 STRIKER Lv.40/R2 두 유닛을 다르게 키운 결과:

| 빌드 | 구성 | HP | DPS | 스테이지12 |
|---|---|---|---|---|
| A 글래스캐논 | 광폭+정밀, atk·crit 각인 | 3090 | 2716 | 3.2초 클리어 |
| B 브루저 | 요새+흡혈, hp·def 각인 | 4720(실효6136) | 1473 | 5.9초 클리어 |

> 출발점이 완전히 같아도, 투자 방향이 유닛의 정체성을 바꾼다.

### 3-5. 소환(가차) — 유닛 획득 (`gacha.mjs` + `rng.mjs`)

- **등급 확률**: N 50 / R 33 / SR 14 / SSR 3 (가중치). 등급이 시작 랭크를 정함(전설=R3).
- **역할**: VANGUARD/STRIKER/SUPPORT 균등.
- **천장(pity)**: 90회 안에 SSR 보장, SSR 획득 시 리셋.
- **10연차**: 최소 1개 SR 이상 보장.
- **RNG 주입**: `makeRng(seed)` 로 재현 가능 → 확률 검증·밸런싱에 사용.

### 3-6. 장비 — 착용 성장 (`gear.mjs`)

- 슬롯 3종: **weapon(공격) · armor(생존) · accessory(효과/속도)**.
- 장비는 flat 스탯 + 전투 효과(치명/흡혈/관통)를 준다 → modifiers의 한 소스.
- **제작(craft)** → 인벤토리, **장착(equip)** → 슬롯, **강화(enhance)** → flat +12%/레벨.
- 강화(각인)이 "본체 %투자"라면, 장비는 "착용 flat + 효과"로 역할이 갈린다.

검증(`node system/demo-gacha.mjs`): 10연차(천장·바닥보장 동작) 후,
뽑은 유닛에 장비 3종 장착·강화 → 전투력 347 → 679.

### 3-7. 캐릭터 정체성 + 전용 스킬 — "이 캐릭터가 누구인가"

§3-1~3-6이 "같은 유닛을 어떻게 키우나"(기계적 차별화)라면, 이건 "캐릭터가 누구인가"
(정체성 차별화). 아키텍처 원칙을 지켜 **정체성은 Concept, 전투는 Core**로 분리한다.

- **유닛** = `archetype`(역할·전투, Core가 앎) + `characterId`(정체성, Concept가 정의)
- **캐릭터 도감**은 Concept의 `roster` 배열: `{ id, name, emoji, title, personality,
  element, archetype, rarity, signature }`. Core는 mechanical 필드(archetype/
  rarity/signature)만 읽고, 이름·성격·속성 flavor는 표시에만 쓴다.
- **전용(시그니처) 스킬**: 캐릭터마다 고유 스킬 1개. 일반 슬롯이 아니라 **항상 발동**하며
  **랭크에 비례해 강해짐**(정체성=성장). `SKILL_CATALOG`의 `signature:true` 항목.
- **소환**은 도감 pool을 주입받아 **개별 캐릭터**를 뽑는다:
  `summonMulti(state, n, rng, concept.roster)`. pool을 안 주면 원형만 뽑는 기존 동작
  (하위호환 — 데모/시뮬 불변).
- **컨셉 스왑**: 같은 캐릭터 슬롯이 판타지=카엘(불꽃의 검사), SF=유닛-07(화염 프레임)로.
  **같은 시스템·같은 mechanical, 다른 인물**.

> "IP=시스템" 원칙 유지: 캐릭터 개성은 Concept로 들어가므로, Core는 여전히 이름을 모른다.

### 3-7-b. 친밀도 + 대사/감정 — "말하는 캐릭터"

정체성이 "누구인가"라면, 친밀도는 **애착 루프**다. `core/intimacy.mjs`:
- 유닛별 `intimacy` 포인트, **선물**(currency 소모)로 상승, 레벨당 전 스탯 +2%
  (modifier 파이프라인에 합류) — 키우는 캐릭터가 나만의 강함이 됨.
- 대사는 Concept roster의 `lines`(greet/bond/levelup) — 캐릭터별·컨셉별. 판타지
  카엘 "가자, 전부 태워버리자!" ↔ SF 유닛-07 "타겟 확인. 발열 개시."
- 앱: 캐릭터 화면에 **말풍선** + 친밀도 게이지 + 선물 버튼. 선물 시 인사→유대 대사로
  전환, 레벨업 시 레벨업 대사.
- **코스튬**: 친밀도 Lv로 해금하는 외형 변경 + 소량 보너스(전 스탯 %). 코스튬 데이터
  (이름·외형·보너스)는 Concept, 장착 시 보너스를 유닛에 세팅→Core 모디파이어가 읽음.
  장착하면 표시 이모지(외형)가 바뀐다. 판타지 "홍염 예복"👘 ↔ SF "강화 프레임"🦾.

### 3-8. 속성 상성 — 기계적이면서 정체성

`core/elements.mjs`: 고정 속성 5종(FIRE/WATER/WOOD/LIGHT/DARK)과 상성표.
FIRE>WOOD>WATER>FIRE 순환 + LIGHT↔DARK. 유리 ×1.3, 불리 ×0.8.

- 유닛은 `element`(Core ID)를 갖고, 스테이지도 속성을 순환 배정(`getStage`).
- 판정 시 각 유닛의 dps에 `affinity(유닛속성, 적속성)` 배수 적용 → 카운터 속성을
  데려가면 더 빨리 클리어.
- 속성 없는 유닛(데모/시뮬)은 상성 1.0 → **하위호환**.
- 표시명은 Concept가 매핑: FIRE = 판타지 "불" / SF "열" (같은 ID, 다른 이름).

검증: 동일 STRIKER가 WOOD 적 상대로 FIRE(유리) DPS 1091 / 무속성 840 / WATER(불리)
672 = 정확히 ×1.3 / ×1.0 / ×0.8.

## 4. 레이어별 스왑 규칙

| 바꾸는 것 | 건드리는 파일 | core 변경 | 예시 |
|---|---|---|---|
| **장르** | `genres/*.mjs` 추가/교체 | ❌ 없음 | RPG → 방치형 |
| **컨셉** | `concepts/*.mjs` 추가/교체 | ❌ 없음 | 판타지 → SF |
| **밸런스** | `core/*` 상수 | ⭕ 공통 반영 | 성장률, 적 강화율 |

새 장르(예: 실시간 PvP)나 새 컨셉(예: 좀비 아포칼립스)을 붙일 때
**core 는 한 줄도 수정하지 않는다.** 파일 하나만 추가하면 된다.

## 5. 검증된 동작

**`node system/demo.mjs`** — 장르·컨셉 스왑
1. **컨셉 스왑** — 판타지/SF 로 렌더. 스탯 숫자 **완전 동일**, 이름만 다름.
2. **장르 스왑** — RPG(능동 4층) / 방치형(2시간 오프라인 → 7섹터 자동 파밍·환생)

**`node system/demo-build.mjs`** — 캐릭터 성장 분화
- 동일 STRIKER 두 유닛을 레벨업·돌파·스킬·각인으로 다르게 육성 → 성능 분화(§3-4)

**`node system/demo-gacha.mjs`** — 소환 + 장비
- 시드 10연차(천장·바닥보장) → 뽑은 유닛에 장비 제작·장착·강화(§3-5, §3-6)

## 6. 폴더 구조

```
system/
├── core/            # IP — 고정 자산 (장르/컨셉 무관)
│   ├── balance.mjs      # 밸런스 상수 단일 소스 (시뮬레이터가 튜닝)
│   ├── archetypes.mjs
│   ├── stats.mjs        # 성장 공식 + 모디파이어 반영
│   ├── modifiers.mjs    # 성장요소 합산 파이프라인
│   ├── skills.mjs       # 스킬 카탈로그/장착/강화
│   ├── enhance.mjs      # 강화(각인) 노드
│   ├── gear.mjs         # 장비 카탈로그/제작/장착/강화
│   ├── gacha.mjs        # 소환(확률·천장)
│   ├── rng.mjs          # 시드 가능 난수
│   ├── character.mjs    # 레벨업·돌파·장착·강화 액션
│   ├── units.mjs
│   ├── resolution.mjs   ← 판정 엔진 (심장)
│   ├── progression.mjs
│   ├── economy.mjs
│   └── gameState.mjs
├── genres/          # Driver — 스왑 (rpg / idle)
├── concepts/        # Skin — 스왑 (fantasy / scifi / index)
├── sim/             # 밸런스 시뮬레이터
│   ├── autoplayer.mjs   # 합리적 유저 투자 전략
│   ├── balance.mjs      # 7일 시뮬레이션 + 병목/게이트/튜닝 실험
│   └── report.mjs       # 곡선 시각화 HTML 리포트 생성
├── demo.mjs         # 장르·컨셉 스왑 데모
├── demo-build.mjs   # 캐릭터 성장 분화 데모
└── demo-gacha.mjs   # 소환 + 장비 데모
```

## 8. 밸런스 시뮬레이터 (`system/sim/`)

문서가 "미정"으로 남긴 **숫자 감각**을 코드로 검증한다. 합리적 오토플레이어(자원을
벌어 재투자하는 가상 유저)를 7일간 돌려 성장 곡선·병목·해금 페이싱을 리포트한다.

- **`node system/sim/balance.mjs`** — 7일 곡선 + 병목 검출 + 해금 게이트 페이싱 +
  튜닝 실험(밸런스 상수를 바꿔 재시뮬레이션 비교)
- **`node system/sim/report.mjs`** — 곡선을 자체완결 HTML(인라인 SVG)로 시각화

### 진단 → 수정 → 재검증

1. **지수적 벽(진단)**: Day1에 ~30층 폭발 후 하루 +1. 파워가 **레벨 상한(랭크×20)**에
   묶이고, 유일한 돌파 수단인 환생이 **수입만** 늘리고 파워는 안 늘려 1.14ⁿ 난이도를
   못 따라감 → 최저달성률 45%로 뒤처짐, 병목 6회.
2. **환생이 장식용이었다(원인)**: `state.prestige`가 포인트만 쌓고 아무 배수도 미적용.
3. **수정**: 환생에 **상한 없는 글로벌 파워 배수**(`prestigePowerBonus`) 추가 +
   환생 시 회차 리셋(진행도 `peakStage`는 유지)하는 정통 방치형 루프.
   벽에서 환생 → 배수로 더 깊이 재등반.
4. **재검증**: 벽 해소. 36층→53층, 최저달성률 45%→91%, 병목 6회→**0회**(급정체 없음).

> 시뮬레이터가 문제를 찾고 → Core를 고치고 → 같은 시뮬레이터로 해소를 확인하는
> 폐루프. 아트 한 장 그리기 전에 밸런스가 굴러가는지 증명한다.

### 지표 정교화 (resolve 기반 headroom)

옛 "달성률"은 `hp·atk·def` 가중합 **프록시**라 실제 전투와 스케일이 달라 왜곡됐다 —
유물·펫 추가 후 200%+로 폭증해 "과강"처럼 보인 게 **거짓 경보**였다. `resolve`로 직접
**"이길 수 있는 최심 스테이지 − 파밍 스테이지 = 전투 여유(headroom)"** 를 측정하도록 교체
(`resolution.resolve`가 `margin` 반환, `sim/balance.deepestWinnable`가 이분 탐색).

결과: 계정 배수가 7일 만에 ×30으로 커져도 **전투 여유는 +11~+13층으로 안정** — 파워
인플레이션을 지수 난이도가 그대로 흡수하는 건강한 곡선임이 드러났다.

### 설계 산물: `core/balance.mjs`

모든 밸런스 상수(적/보상/성장/비용/환생)를 **단일 소스**로 모았다. 시뮬레이터가
`withBalance(override, fn)`로 값을 바꿔가며 실험하고, 게임 코드는 이 값을 참조만 한다.
새 성장·비용 곡선을 붙여도 한 곳만 수정하면 전 시스템에 반영된다.

## 9. 세이브 & 영속성 (`core/save.mjs`)

`gameState`는 함수·순환참조 없는 **순수 데이터**라 JSON으로 그대로 왕복한다.

- `serialize(state)` / `deserialize(json)` — 버전(`SAVE_VERSION`) 게이트로 구버전 거부,
  누락 필드 기본값 보정(`normalize`), uid 시퀀스 동기화(`syncSeq`)로 로드 후 신규 생성
  충돌 방지.
- **오프라인 정산**: `idleGenre.collectOffline(state, now)`가 `lastTick` 이후 경과분을
  방치 틱으로 계산(상한 8시간) → 재접속 시 "다시 오셨네요" 보상 팝업.
- **앱 연동**(`app/storage.js`, `app/useGame.js`): 웹 localStorage에 매 틱 자동 저장 +
  창 닫힘/숨김 시 저장. 로드 시 오프라인 보상 표시, 헤더에 초기화 버튼.

검증: 11유닛 세이브 왕복 시 전투력·천장·지갑 보존, 새 uid 무충돌, 버전 불일치 거부;
2시간 공백 → 오프라인 보상 정산·팝업 표시(브라우저 실측).

## 10. 콘텐츠 & 계정 성장

**일일 콘텐츠**(`core/daily.mjs`) — D1/D7 리텐션 루프, 자원 faucet(특히 소환권):
- **출석**: 7일 순환 보상, 하루 1회, streak 유지 (UTC epoch-day 기준)
- **일일 미션**: 소환/강화/던전 카운트 → 보상. 하루 리셋. 액션에서 `recordMission` 호출
- **던전**: 자원별(골드/정수) 즉시 대량 파밍, 하루 입장 제한

**계정 성장 축** — 모두 `accountMods(state)`로 합산되어 전투(`resolve` powerMult)와
방치 수입(`idle.tick` currency/growthMult)에 곱해진다:
- **유물**(`core/relics.mjs`): 3종 영구 배수, currency로 즉시 강화(제자리). 장기 싱크.
- **펫**(`core/pets.mjs`): 수집 축. 다이아로 소환(등급 확률), **중복은 레벨업(합성)**,
  최대 3마리 장착 → 장착 펫만 보너스. 다이아 소비처로 BM과 연결.

> `accountMods`가 **환생 + 유물 + 펫**을 한 곳에서 곱한다. 새 계정 성장 축을 붙일 때도
> 여기 한 줄만 추가하면 전투·수입에 자동 반영 — 실제로 펫이 이 방식으로 붙었다.
> (순환 의존 방지: 펫은 gacha가 아니라 자체 등급표를 씀)

앱: `콘텐츠` 탭(출석·미션·던전·**펫**·유물). 하위호환 — 구버전 세이브는 `normalize`가 기본값 주입.

## 11. BM / 상점 골격 (`core/shop.mjs`)

프리미엄 재화 **gem**(economy에 추가)과 3종 구매 경로:
- **광고(ad)**: 무료, 하루 횟수 제한(`state.daily.ads`). 골드/소환권/다이아 지급
- **다이아(gem)**: gem 소모 → 소환권·골드·정수 대량
- **패키지(package)**: 실결제(모의). 스타터=1회성(`state.shop.purchased`), 성장/월정액
- `grant`의 `*Stage` 키는 진행도(peakStage) 보상에 비례 스케일 → 후반에도 가치 유지.

> 결제 SDK는 미연동(골격). 버튼은 보상 흐름 시연용 모의 지급. 실 연동 시 `purchase()`의
> package 분기만 결제 콜백 뒤로 옮기면 된다.

앱: `상점` 탭, 상단 자원바에 다이아 표시. 컨셉이 gem 표시명 매핑(판타지 "다이아"/SF "퀀텀").
