## 1. 현재 상태 진단 (있는 그대로)

**기존 게임 axdata_01 "eldria-idle"은 이미 크고 성숙하다.** 리뉴얼은 "빈 땅에 새로 짓기"가 아니라 "잘 지은 집을 리모델링하기"다.

- **코어**: 가챠·장비(+carry/salvage)·경제·투기장·길드·캠페인·강화·문장·코스튬·정령·수호신·IAP·일일·이벤트 등 40개+ 모듈. 순수 ESM이라 서버 재사용 가능.
- **앱(Expo)**: 화면 15개(방치·로스터·가챠·상점·투기장·성장·전투뷰·온보딩·메타·우편…), 백엔드(Supabase 클라우드 동기화·PvP), 다국어(i18n), IAP, 스프라이트 애니메이션, 빌드 파이프라인(EAS APK/AAB·OTA).
- **아트**: PNG 166장 + 렌더 파이프라인(21종 KayKit 캐릭터, out_roster_full 2688장 준비).
- **이미 존재하는 설계 문서**: `redesign-plan.md`(코어 옵션화), `ART_STRATEGY.md`(저비용 고퀄 3레인), `ROGUELITE-PLAN.md`(원정 모드), `PHASE2_DESIGN.md`(실 PvP·시즌), `ART_BIBLE.md`, `ART_PIPELINE_3D.md`.

**결론**: 게임 규칙(코어)의 "양"은 이미 충분하다. 부족한 것은 **체감 퀄리티** — 눈에 보이는 아트, 손에 닿는 UI/UX, 처음 5분의 인상, 그리고 콘텐츠의 "느낌". 리뉴얼의 힘은 여기에 집중해야 한다.

## 2. "퀄리티"를 4개 축으로 정의

막연한 "퀄 높이기"를 검증 가능한 축으로 쪼갠다. 각 축은 독립적으로 좋아질 수 있다.

| 축 | 무엇 | 지금 상태 | 활용 시스템 |
|---|---|---|---|
| **A. 아트/비주얼** | 캐릭터·몹·이펙트·배경·아이콘 | 3D 프리렌더 파이프라인은 있으나 게임에 완전 반영·통일 안 됨 | 05 렌더 + 09 스튜디오 + 렌더 팩 |
| **B. UI/UX·연출** | 화면 레이아웃·전투 연출·피드백·애니·사운드 | 기능은 다 있으나 "예쁨/맛"이 약함 | app/theme·motion·BattleView·feedback |
| **C. 게임성·리텐션** | 첫인상·성장 페이스·콘텐츠 루프 | 모듈은 많으나 방향/페이스 산만 가능성 | 코어 + redesign-plan·ROGUELITE-PLAN |
| **D. 기술·완성도** | 성능·버그·빌드·저장 안정성 | 테스트 있음, 회귀 관리 필요 | 코어 테스트 + Expo 빌드 |

## 3. 축별 개선 항목 (우선순위 매김)

우선순위 기준: **체감 임팩트 ÷ 노력**. 🔴=먼저, 🟡=다음, 🟢=나중.

### A. 아트/비주얼
- 🔴 **A1. 3D 프리렌더 전 캐릭터 반영** — out_roster_full(21종×16동작) 렌더를 게임 규격(`assets/char/<concept>/<id>.png`)으로 전량 배선. 이미 준비된 자산을 "쓰기만" 하면 됨. → `ART_STRATEGY` 레인 A.
- 🔴 **A2. 렌더 품질 튜닝 1회 셋업** — Blender 3점 라이팅+림라이트+셀셰이더+외곽선+SSAA(2배 렌더 후 다운샘플). 파이프라인 설정만 바꿔 전 캐릭터 퀄 상승. → `ART_STRATEGY` §2. (PC/GPU 작업)
- 🟡 **A3. 로컬 AI 스타일 패스(레인 B)** — 3D 렌더를 밑그림으로 ComfyUI img2img+ControlNet로 화풍만 입힘. 장당 과금 0, 일관성 유지. → `ART_STRATEGY` §3. (PC/GPU 작업)
- 🟡 **A4. UI 아이콘·이펙트 통일** — 아이콘(uiIcons)·타격/스킬 이펙트 톤 통일. 09 스튜디오로 UI 팩 생성 가능.
- 🟢 **A5. 대표 키비주얼(레인 C, 유료 소량)** — 스토어/광고용 히어로 아트 소수만 유료 AI.

### B. UI/UX·연출
- 🔴 **B1. 디자인 토큰 정비** — `app/theme.js` 색·타이포·간격·라운드·그림자 토큰 재정의. 모든 화면이 일관된 룩을 갖도록 기반부터. (모바일 우선)
- 🔴 **B2. 전투 연출 강화** — `BattleView`에 타격 히트스톱·데미지 팝업·스킬 컷인·승리 연출. 방치형은 전투가 "보는 맛"이라 임팩트 큼.
- 🟡 **B3. 촉각·사운드 피드백** — expo-haptics + expo-av로 버튼/획득/레벨업/가챠 연출 강화. `feedback.js` 확장.
- 🟡 **B4. 첫 5분(온보딩) 리디자인** — `Onboarding`·첫 가챠·첫 전투의 감정 곡선 설계. 리텐션 직결.
- 🟢 **B5. 화면별 마이크로 인터랙션** — 리스트 진입 애니, 스켈레톤 로딩, 트랜지션.

### C. 게임성·리텐션
- 🔴 **C1. 코어 정리(옵션화) 완성** — `redesign-plan.md`의 features 플래그로 "단순 코어 + 선택 모듈" 확정. 산만함을 줄이고 페이스를 잡는 게 리뉴얼의 뼈대. (비파괴·가역)
- 🟡 **C2. 원정(로그라이트) 모드** — `ROGUELITE-PLAN.md` 그대로 얇게 얹기. 2026 흥행 공식(방치+수집+오토배틀러+로그라이트 런). 전투 재사용, 신규 코드 최소.
- 🟢 **C3. 실 PvP·시즌(Phase 2)** — `PHASE2_DESIGN.md`. 서버 필요, 리텐션 후반 레버. 로컬 폴백 유지.

### D. 기술·완성도
- 🔴 **D1. 회귀 안전망 유지** — 모든 단계에서 `node --test system/test/*.test.mjs` + 스모크 통과를 게이트로. 코어 안 깨고 리뉴얼.
- 🟡 **D2. 성능·저장 점검** — 방치 틱 루프·리스트 렌더 성능, 세이브 하위호환.
- 🟡 **D3. 빌드 검증** — `build:play`(자립형 HTML)로 매 단계 웹 프리뷰, 최종 APK/AAB.

## 5. 단계별 실행 로드맵 (승인 후)

각 단계 = **검증 가능한 목표 + 확인 방법**. 비파괴·가역 원칙(회귀 테스트 항상 통과).

| 단계 | 내용 | 확인 방법 |
|---|---|---|
| **S0** | 기존 게임 실제 구동·현물 점검(웹 프리뷰), 약점 스크린샷 목록화 | play.html/web 구동, 화면별 캡처 |
| **S1 (B1·D1)** | 디자인 토큰 정비 + 회귀 게이트 확립 | 테스트 통과 + 전 화면 룩 일관 |
| **S2 (A1·A2)** | 3D 렌더 전 캐릭터 반영 + 렌더 품질 튜닝(PC) | 로스터/전투에 신규 스프라이트 표시 |
| **S3 (B2·B3)** | 전투 연출·피드백 강화 | 웹 프리뷰로 전투 1회 "맛" 체감 |
| **S4 (C1)** | 코어 옵션화 완성(단순↔풀 모드) | 단순 모드 end-to-end + 풀 모드 회귀 |
| **S5 (B4·A4)** | 온보딩 리디자인 + UI 톤 통일 | 첫 5분 플레이 흐름 확인 |
| **S6 (C2)** | 원정(로그라이트) 모드 | 1런 완주 + run.test 통과 |
| **S7 (A3·D3)** | 로컬 AI 스타일 패스 + 빌드 검증 | 스타일 적용 전후 비교, APK 빌드 |
| **S8+** | (선택) C3 실 PvP·시즌 | 서버 연동, 로컬 폴백 유지 |

> 🔴 항목(A1·A2·B1·B2·C1·D1)까지가 "1차 리뉴얼"의 핵심. S1~S4가 체감 퀄을 가장 크게 올린다.

## 7. S0 현물 점검 결과 (2026-07-18)

기존 게임(axdata_01)의 웹 빌드(`dist/`)를 로컬 서버로 띄워 실제 구동 확인 + UI 소스 정독. (스크린샷은 방치형 렌더 루프가 CDP를 점유해 이 원격 환경에서 캡처 불가 → 시각 캡처는 Gim PC에서. 구조는 접근성 트리로, 시각 품질은 소스로 판단.)

**구동 상태**: 정상. 제목 "엘드리아 연대기", 탭 5개(전투/영웅/소환/콘텐츠/상점), 방치 전투 루프·무한탑(층)·온보딩 동작. 콘솔 에러 없음.

**프레젠테이션 레이어 진단 (리뉴얼이 건드릴 코드)**

| 파일 | 규모 | 상태 | 리뉴얼 시사점 |
|---|---|---|---|
| `theme.js` | 45줄 | 색·SPACE·RARITY 토큰 있음. 단 **타이포/라운드/그림자(elevation) 토큰 없음** | B1: 토큰 확장(폭 넓히기)이 정확한 첫 코드 작업 |
| `motion.js` | 11줄 | reduce/eco 플래그뿐, **애니메이션 프리미티브 없음** | B2·B5: 이징·트랜지션 헬퍼를 새로 만들어야 |
| `feedback.js` | 121줄 | 사운드 신스+햅틱 **이미 탄탄** | B3는 거의 완성 — 확장만 |
| `BattleView.js` | 193줄 | 스프라이트·HP바·러지·데미지숫자 **기본 연출 있음** | B2: 히트스톱·화면흔들림·스킬컷인·승리연출 **없음** → 여기가 임팩트 큼 |
| `RosterScreen.js` | **1400줄** | 단일 화면 과대 | C1/유지보수: 분해·정리 후보(가장 큰 복잡도 핫스팟) |

**핵심 결론 3가지**
1. 디자인 토큰 기반은 있으나 얇다 → **B1(토큰 확장)이 저노력·고효과 첫걸음**.
2. 전투 연출은 뼈대만 있고 "타격감"이 없음 → **B2가 체감 퀄 최대 레버**(단 캐릭터 비주얼은 아트 A1 의존이라 나중).
3. `RosterScreen.js` 1400줄은 리뉴얼·유지보수의 병목 → C1 코어정리 때 우선 분해 대상.

**S0 기준 권장 1차 코드 착수**: **B1(디자인 토큰 확장) + D1(회귀 테스트 게이트)** → 이어서 B2(전투 연출). 아트(A1·A2)는 Gim PC 렌더 후 합류.
